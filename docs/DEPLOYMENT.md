# Публичный хостинг Zenthril

Краткий чеклист для выкладки **в интернет** (VPS / облако), а не только в LAN.

## 1. Архитектура

Типичная схема:

- **Клиент** — статика Vite (`npm run build` → папка `client/dist`) за reverse proxy с **TLS** (Caddy, nginx, Traefik) на домене вроде `https://app.example.com`.
- **API + WebSocket** — бинарь Go или контейнер на том же или отдельном хосте, например `https://api.example.com` (порты 443 снаружи, внутрь — `:8080`).

База **PostgreSQL** и **Redis** не должны быть доступны из интернета — только из приватной сети / localhost сервера с бэкендом.

## GitHub как «хост»

**Только частично.** GitHub хранит код и умеет запускать **Actions** (сборка, тесты), но **не заменяет сервер** для Go, базы и постоянного WebSocket.

| Роль | Подходит ли GitHub |
|------|-------------------|
| Репозиторий + CI | Да |
| Статика веб-клиента (`client/dist`) | Да — **[GitHub Pages](https://docs.github.com/pages)** |
| API, `/ws`, PostgreSQL, Redis | Нет — нужен отдельный хост (VPS, Railway, Fly.io, Render и т.п.) |

Типичная связка: **Pages** отдаёт SPA (например `https://<user>.github.io/<repo>/` или свой домен), **API** — на другом origin. При сборке для Pages задайте `VITE_API_BASE=https://ваш-api.example.com`. В бэкенде в `CORS_ALLOWED_ORIGINS` и `WS_ALLOWED_ORIGINS` укажите origin клиента на Pages (для project site это обычно `https://<user>.github.io` — без пути; путь в URL страницы на CORS не влияет).

Если сайт лежит не в корне домена (`/repo/` на GitHub Pages), в `client/vite.config.ts` нужен **`base: '/имя-репо/'`**, иначе сломаются пути к JS/CSS. Для пользовательского домена в корне — `base: '/'`.

Деплой статики через Actions: соберите `npm run build` в `client/`, загрузите содержимое `dist` в артефакт Pages (официальный гайд: *GitHub Actions → Deploy static site*). Секрет репозитория `VITE_API_BASE` передайте в шаг сборки как переменную окружения.

## 2. Сборка клиента под прод

При другом домене API обязательно задайте origin **без** завершающего слэша:

```bash
cd client
echo 'VITE_API_BASE=https://api.example.com' >> .env
npm ci
npm run build
```

Поведение:

- HTTP-запросы идут на `VITE_API_BASE`.
- WebSocket строится как `wss://` к тому же host, что в `VITE_API_BASE` (путь `/ws`).

Если `VITE_API_BASE` не задан, клиент по-прежнему целится в `текущий хост:8080` (удобно для локальной разработки).

## 3. Переменные бэкенда (обязательно для прода)

Скопируйте корневой `.env.example` → `.env` на сервере и выставьте:

| Переменная | Продакшен |
|------------|-----------|
| `DB_URL` | Надёжный пароль, `sslmode=require` если БД с TLS |
| `JWT_SECRET` | Длинная случайная строка (не из примеров в репозитории) |
| `CORS_ALLOWED_ORIGINS` | Точный origin веб-клиента, например `https://app.example.com` |
| `WS_ALLOWED_ORIGINS` | Обычно те же значения, что и для CORS |
| `ADMIN_USER_IDS` | UUID операторов через запятую; пусто — без админ-роутов |

Пустые `CORS_ALLOWED_ORIGINS` / `WS_ALLOWED_ORIGINS` в коде означают разрешение любого origin — **для публичного сервиса задайте явный список**.

`NODE_DOMAIN` для продакшена должен совпадать с публичным именем узла, если используете федеративные функции.

## 4. Docker Compose из репозитория

Файл `docker-compose.yml` ориентирован на **локальную разработку** (в т.ч. упрощённая настройка Postgres). Для продакшена:

- Задайте сильные пароли БД, уберите `POSTGRES_HOST_AUTH_METHOD: trust`.
- Не публикуйте порты `5432` / `6379` наружу, если бэкенд на той же машине — достаточно внутренней сети Docker.
- Замените `JWT_SECRET` в `environment` на значение из секретов (Docker secrets, env-файл вне git, переменные панели хостинга).

## 5. TLS и reverse proxy

Рекомендуется терминировать TLS на **Caddy** или **nginx** и проксировать на бэкенд:

- Заголовки `X-Forwarded-For` / `X-Forwarded-Proto` — для корректного IP и ссылок (бэкенд уже учитывает `X-Forwarded-For` в защите).
- WebSocket: `Upgrade` и `Connection` должны пробрасываться (в nginx — `proxy_http_version 1.1` и заголовки upgrade).

Если API и приложение на **одном** домене и одном TLS (например `/api` → бэкенд), убедитесь, что маршруты и префиксы совпадают с тем, что ожидает клиент (сейчас клиент ждёт пути вида `/api/v1/...` от **корня** origin из `VITE_API_BASE`).

## 6. Резервное копирование

- Регулярный **`pg_dump`** (или снапшоты тома PostgreSQL) + проверка восстановления.
- Храните бэкапы **вне** того же сервера, что прод.

## 7. Обновления

- Останавливайте бэкенд → миграции (если добавите отдельный шаг) → новый бинарь/образ → старт.
- Клиент: пересборка с тем же `VITE_API_BASE`, выкладка нового `dist`.

## 8. Юридическое и безопасность

- Заполните [PRIVACY.md](PRIVACY.md) и контакты в [SECURITY.md](../SECURITY.md).
- Включите на GitHub приём отчётов об уязвимостях (Security).

## 9. Мониторинг

- Проверка `GET /health` (или ваш health-эндпоинт) с внешнего uptime-сервиса.
- Логи контейнера/сервиса и алерты по диску и памяти.
